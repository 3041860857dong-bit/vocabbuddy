## 〇、估算假设
- 地域：中国大陆标准地域（广州 / 上海 / 北京，三者同价；不推荐境外地域，对国内用户延迟高且更贵）。
- 部署：1 台 CVM 跑 Docker Compose（FastAPI 应用容器，DB 用托管 TencentDB，不放在容器里）。
- 业务：读多写少，词库几 MB、用户数据小，日活几十到几百。
- 计费：默认**包年包月**（月付口径，年付通常再 8~8.5 折 ≈ 省 15%）；带宽默认"按带宽包"，小流量场景也给出"按流量"备选。
- 货币：人民币 / 月。

## 一、CVM 选型（三档）

| 档位 | 适用场景（日活假设） | 推荐机型系列 | vCPU/内存 | 系统盘 | 公网带宽建议 |
|---|---|---|---|---|---|
| **入门档** | 个人 / 自测 / 小圈子，日活 ≤ 50，峰并发个位数 | SA2 或 SA5（AMD，性价比高）；备选 Lighthouse 轻量 | 2核4GB | 50GB 通用型SSD云硬盘 | 按流量（¥0.8/GB）或 1–2Mbps 带宽包 |
| **标准档** | 小团队 / 班级 / 公开试用，日活 100–500，峰并发 20–50 | SA5 或 S5（Intel） | 4核8GB | 100GB 通用型SSD | 3–5Mbps 带宽包或按流量 |
| **推荐档** | 成长型 / 多班级，日活 500–2000，峰并发 50–150 | S5 / S6 / SA5 | 4核16GB（或 8核16GB） | 200GB 通用型SSD | 8–10Mbps 带宽包 |

**关键说明：**
- **SA5/SA2（AMD）比同规格 S5/S6（Intel）便宜约 10–20%**，性能对读多写小的 Web 应用完全够，优先推荐 SA5。
- **轻量应用服务器（Lighthouse）** 是套餐式（vCPU+内存+带宽+系统盘+公网IP 打包），入门档若只跑这一单应用，Lighthouse 2核4GB（含 4–6Mbps 带宽 + 60–80GB SSD）比单独买 CVM+带宽更省，可作为最省备选。但 Lighthouse 不适合后续做复杂组网/多服务扩展。
- 内存给到 4GB 起步是因为 Docker + FastAPI + nginx 三个进程 + 系统本身，2GB 偏紧、易 OOM。
- 系统盘用**通用型SSD（GA2/增强型SSD 亦可）**，别用普通云硬盘，IOPS 差。词库/用户数据小，系统盘容量够；未来变大再挂独立云硬盘数据盘并快照。

## 二、TencentDB for MySQL 选型（三档，版本固定 8.0）

| 档位 | 架构 | 规格（内存/存储） | 高可用方案 |
|---|---|---|---|
| **入门档** | 双节点（高可用版）起步即推荐双节点；预算极紧可选单节点（基础版） | 1核2GB / 50GB SSD | 主备双节点，自动故障切换（单节点无切换） |
| **标准档** | 双节点（高可用版） | 2核4GB / 100GB SSD | 主备双节点 + 自动备份 |
| **推荐档** | 双节点（高可用版） | 4核8GB / 200GB SSD | 主备双节点 + 只读副本预留（读压上来再加 RO） |

**关键说明：**
- **版本锁定 MySQL 8.0**（应用依赖 8.0 特性/默认认证）。
- **强烈建议双节点（高可用版）而非单节点**：单节点便宜约一半但无自动故障切换，DB 挂了要人工恢复，对"用户学习进度"这类有状态数据不友好。预算紧时入门档用单节点过渡，但上线即上双节点最佳。
- 存储 50GB 起步对"词库几MB + 用户数据小"极度富余，预留了成长空间；TencentDB 存储一般随规格含基础额度，超额按 ~¥0.5–1/GB/月计费（已折算进区间）。
- 读多写少特征：日活 2000 以内双节点足够；若未来读压明显上升，加 **1 个只读副本（RO）** 分流查询即可，不必直接升主实例规格。

| 档位 | CVM（含盘+带宽） | TencentDB for MySQL | **合计区间/月** | 最省备选 |
|---|---|---|---|---|
| **入门档** | ~¥205（SA5 2核4GB ¥150 + 50GB SSD ¥30 + 2Mbps ¥25） | 双节点1核2GB/50GB **~¥130**；（单节点仅 ~¥95） | **~¥330–360**（双节点DB）；单节点DB → **~¥300–330** | Lighthouse 套餐 ¥80–120 + 单节点DB → **~¥200–250** |
| **标准档** | ~¥530（SA5 4核8GB ¥360 + 100GB SSD ¥60 + 5Mbps ¥110） | 双节点2核4GB/100GB **~¥330–430** | **~¥860–960** | — |
| **推荐档** | ~¥950（S5 4核16GB ¥600 + 200GB SSD ¥120 + 10Mbps ¥230） | 双节点4核8GB/200GB **~¥700–950** | **~¥1650–1900** | 若升 8核16GB CVM ≈ +¥150 → ~¥1950 |

**按量计费 vs 包年包月：**
- **按量计费** ≈ 包月单价 × 1.3~2.0，按小时结算，无长期承诺，适合短期验证/压测。
- **包年包月** 月付即有折扣，**年付通常再 8~8.5 折**（约省 15%），稳定长期运行必选。
- **建议**：先用按量跑通验证 1–2 周 → 确认规格合适后转**年付包年包月**最省。

## 四、采购后首次启动 Runbook（要点）

1. **操作系统初始化**：公共镜像选 Ubuntu 22.04 LTS 或 TencentOS Server 3 / CentOS Stream。配置 SSH 密钥登录 + 禁用密码登录；安全组只开 22(SSH)、80(HTTP)（后续 HTTPS 再加 443）；SSH 建议改非 22 端口，配 Fail2ban。设时区 Asia/Shanghai + NTP 时间同步（DB 与应用时间一致很重要，SRS 依赖时间戳）。
2. **安装 Docker + Docker Compose**：装 docker-ce + docker-compose-plugin（Compose v2，`docker compose` 子命令）。启 docker 服务 + 开机自启；验证 docker version / docker compose version。
3. **传项目到服务器**：scp/rsync 整个项目目录（Dockerfile、docker-compose.yml、nginx.conf、.env 模板、data.js/词库、systemd 服务文件、DEPLOYMENT.md）到 /opt/vocabbuddy。注意 .env 不要带真实密钥提交，传模板，服务器上填真实值。
4. **配置 .env 连接 TencentDB（关键）**：填 TencentDB 内网地址（控制台"内网地址"那一栏，不是公网地址）、端口 3306、库名、用户名、密码。TencentDB 白名单必须放 CVM 的内网 IP（或同 VPC 安全组），不要长期开 0.0.0.0/0。确认应用容器通过内网地址连 DB（走内网不收公网流量费、延迟低、安全）。
5. **docker compose up**：先 docker compose build → docker compose up -d。查 docker compose ps / docker logs 确认容器起得来。
6. **导入词库 / 初始化库表**：跑词库导入脚本，把词库写入 TencentDB。验证数据已落库。
7. **验证 health（端到端）**：curl http://<公网IP>:8000/health 应返回 ok。看日志无报错；做一次真实操作：查词 → 背单词（SRS）→ 看学习统计，全链路通。
8. **（可选）systemd 守护**：用项目自带 systemd 服务文件管理 compose 开机自启与异常重启。

## 五、风险提示（采购/上线前必读）

- **公网 IP + HTTP 明文**：账号密码、SRS 学习进度在传输中可被窃听/篡改。尽快上 HTTPS（Let's Encrypt 免费证书，nginx 加 443）。HTTP 阶段不要传敏感信息。
- **TencentDB 白名单务必只放 CVM 内网 IP**：临时调试开了 0.0.0.0/0 后必须收紧，否则 DB 暴露在公网有被拖库风险。
- **容器连 DB 用内网地址**：公网地址不仅不安全，还会被收公网流量费且延迟高。
- **安全组最小权限**：只开 22/80/443；SSH 改端口 + 密钥 + Fail2ban；应用端口（如 8000）不要对公网开放，只由同机 nginx 反代。
- **数据备份**：开启 TencentDB 自动备份（默认保留 7 天，建议确认并适当延长），并做跨地域/对象存储备份。
- **成本护栏**：按流量计费时设带宽上限与流量告警，防止异常流量爆账单；长期跑务必转年付包年包月。
- **单节点 DB 无高可用**：故障需人工恢复，生产环境建议直接双节点起步。
- **监控告警**：CVM 与 TencentDB 都开启云监控基础告警（CPU/内存/磁盘/连接数/慢查询），避免静默故障。
